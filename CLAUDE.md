# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

DentalSys ā€” a multi-tenant SaaS MVP for managing a dental clinic (patients, appointments, clinical treatments, billing, and a calendar/odontogram). Backend: Flask (Python) + PostgreSQL. Two separate Angular 18 frontends share that one backend: `frontend/` (port 4200, used by clinic staff ā€” patients/appointments/billing/etc., gated by the RBAC described below) and `admin-frontend/` (port 4300, used only by the SaaS operator to manage clinics/subscriptions ā€” see "Platform administration" below). UI strings, route labels, and seed data are in Spanish.

The root `README.md` (rewritten 2026-07-11 to match current multi-tenant architecture) is the general project overview; this file is the deeper, gotcha-level guide for anyone (human or agent) working on the code ā€” keep both in sync when a change touches architecture, not just this file.

## Working agreement

**Never `git commit` or `git push` automatically.** All commits and pushes are done manually by the user ā€” staging/committing/pushing is never an implicit step of finishing a task, even after a working fix, even right before reporting completion. Prepare and describe changes; wait to be explicitly asked before committing or pushing.

## Commands

### Docker (primary workflow)
```bash
docker compose up -d --build       # starts backend (:5000), frontend (:4200), admin-frontend (:4300) ā€” no local db container; points at a shared Supabase project instead, see "Deployment" below
docker compose logs -f backend     # tail backend logs
docker compose exec backend flask db migrate -m "message"   # create a migration
docker compose exec backend flask db upgrade                # apply migrations
docker compose exec backend flask seed                      # seed clinic #1 demo data + default subscription tiers
docker compose exec backend flask create-clinic --name "..." --admin-email "..." --admin-password "..."   # defaults to the "Trial" tier, 30-day plan_expires_at ā€” see "SaaS subscription lifecycle" below
docker compose exec backend flask create-platform-admin --email "..." --password "..."   # bootstrap a SaaS-operator user (is_platform_admin=True, no clinic)
```
`backend/entrypoint.sh` runs migrations (as the `dental_user` superuser-ish role, via `MIGRATIONS_DATABASE_URL`) then execs `gunicorn` (as the restricted `dental_app` role, via `DATABASE_URL`) automatically on container start ā€” see "Multi-tenancy" below for why two DB roles exist. **It does NOT run `flask seed`** (removed 2026-06-22) ā€” seeding/demo data is always a deliberate, manual step (`docker compose exec backend flask seed`), never triggered by a container start/restart/crash-recovery, so a restart can never silently inject demo users/patients into real tenant data. Gunicorn has no `--reload` flag here (tuned to behave like prod even locally), so it loads every backend module once at boot: any `.py` change (routes, models, middleware) needs `docker compose restart backend` to actually take effect, even though the file is bind-mounted and visible inside the container immediately. `WEB_CONCURRENCY`/`WEB_THREADS`/`DB_POOL_SIZE`/`DB_MAX_OVERFLOW` are tunable via env vars (defaults are conservative, sized for a free-tier instance) ā€” see `backend/.env.example` for the reasoning and suggested values per Render/Supabase tier combination; bump threads before workers on a tier whose RAM didn't increase, since a worker is a full extra process and a thread is nearly free by comparison.

**Gotcha (found 2026-07-11): stop the containers before any git operation that touches many files at once** (`git checkout <branch>`, a large `git merge`) ā€” `docker compose stop`, do the git operation, `docker compose up -d` after. Docker Desktop's bind-mount file watchers (`ng serve`'s hot-reload especially) can hold locks on files inside `backend/`, `frontend/`, `admin-frontend/` and make `git checkout` hang on a Windows-only *"Deletion of directory '...' failed. Should I try again?"* prompt, or leave files showing as `deleted` in `git status` afterward even though HEAD still has them. This looked like a OneDrive-sync problem at first (the repo also lives inside a OneDrive-synced folder) ā€” pausing OneDrive did NOT fix it, stopping the containers did. If it happens: confirm no `git.exe` process is still running before touching `.git/index.lock`, then `git restore .` to recover (safe ā€” these are unstaged phantom deletions, not real uncommitted work).

### Backend (Flask), run outside Docker
```bash
cd backend
python3 -m venv venv && source venv/bin/activate   # venv\Scripts\activate on Windows
pip install -r requirements.txt
cp .env.example .env        # then edit DATABASE_URL etc.
flask db upgrade
flask seed
flask run --port 5000
```
There is no automated test suite (no pytest files) and no linter configured for the backend ā€” verify changes by exercising the API (e.g. via `/api/docs/`, the Swagger UI) or `flask shell`.

### Frontend (Angular), run outside Docker
```bash
cd frontend
npm install
npm start          # ng serve --port 4200
npm run build       # ng build
npm test            # karma/jasmine ā€” no spec files currently exist in the repo
```
No ESLint config is present. `admin-frontend/` is a separate Angular app/`package.json` with the same commands, served on port 4300 (`npm start` there runs `ng serve --port 4300`).

**Live UI verification when there's no host-side browser available**: install a real browser inside the already-running `frontend` container and drive it with Puppeteer rather than guessing from source ā€” `docker compose exec frontend sh -c "apk add --no-cache chromium && npm install --no-save puppeteer-core"`, then launch with `puppeteer.launch({ executablePath: '/usr/bin/chromium-browser', headless: 'new', args: ['--no-sandbox', '--disable-setuid-sandbox', '--host-resolver-rules=MAP localhost:5000 backend:5000'] })`. The `--host-resolver-rules` flag matters: `environment.ts`'s `apiUrl` is `http://localhost:5000`, which only resolves correctly in a real browser on the host ā€” from *inside* the frontend container, `localhost:5000` is the frontend container itself, so API calls get `ERR_CONNECTION_REFUSED` without that flag redirecting them to the `backend` service. `window.ng.getComponent(domElement)` (Angular's dev-mode debug API, available since `environment.ts` has `production: false`) is useful for inspecting a component instance's actual runtime property values directly ā€” caught a real bug (see the `withComponentInputBinding()` gotcha above) that wasn't visible from the DOM or source alone.

### Single migration / one-off DB script
Migrations live in `backend/migrations/versions/`; create with `flask db migrate -m "..."` then hand-edit if Alembic's autogenerate misses something non-trivial (e.g. enabling RLS, see `a3f9c2d81e47_add_row_level_security.py`, is written by hand with raw `op.execute`).

## Deployment

Two independent environments, each pairing a Render backend + two Render Static Site frontends with its own separate Supabase Postgres project (fully separate databases and Storage buckets, not just config-separated):
- **Production** ā€” tracks `main`. Backend `dental-sys-backend` runs on Render's native Python runtime, NOT Docker (`backend/Dockerfile`/`entrypoint.sh` are never used there ā€” the Start Command is plain `gunicorn ... run:app`, configured directly in Render's dashboard).
- **Testing** ā€” tracks `testing`. Same shape (`dental-sys-backend-testing` + two static sites), a separate Supabase project. `frontend/` and `admin-frontend/` each have a `testing` Angular build configuration + `environment.testing.ts` pointing at the testing backend's URL.

**Local dev shares the testing database.** `docker compose up` has no local Postgres container; `DATABASE_URL`/`MIGRATIONS_DATABASE_URL` in the root `.env` (gitignored, separate from `backend/.env`) point at the testing Supabase project's session pooler. Running `flask seed` or any destructive local action affects the same data the `testing` Render environment serves ā€” there is no disposable local-only database to fall back on.

**Render never runs migrations automatically**, for either environment ā€” no `preDeployCommand`, and prod's native runtime doesn't execute `entrypoint.sh` at all. Every schema change needs a manual `flask db upgrade` against the target environment's schema-owner role, run before/alongside deploying the code that depends on it; prod has drifted behind head this way more than once. When running `flask db` commands manually (outside `entrypoint.sh`'s automatic override), always reproduce its role override explicitly ā€” `DATABASE_URL="$MIGRATIONS_DATABASE_URL" flask db upgrade` locally, or the equivalent prod URL from `backend/.env` for prod ā€” running it bare uses the restricted runtime role, which lacks DDL rights and, on the testing Supabase project specifically, can't even read its own `alembic_version` row (that project auto-enables RLS with no policy on every new table it creates ā€” see the multi-tenancy RLS notes above for the general pattern this falls under). Both Supabase projects use the same two-role split as local (`create_app_role.sql`): a restricted `dental_app` role for runtime, and the schema-owning `postgres` role for migrations only.

## Architecture

### Multi-tenancy: two independent enforcement layers
Every clinic-scoped table (`users`, `patients`, `appointments`, `treatments`, `treatment_plans`, `invoices`, `invoice_items`, `payments`, `payment_plans`, `payment_plan_installments`, `consultorios`, `appointment_types`, `role_permissions`, `subscription_payments`, `treatment_images`) has a `clinic_id`. Isolation between clinics is enforced twice, deliberately:

1. **ORM-level filter** ā€” `backend/app/middleware/tenancy.py`'s `_scoped_models()` list registers a SQLAlchemy `do_orm_execute` session event that injects `clinic_id == g.clinic_id` into every SELECT against those models, including ones triggered by lazy-loaded relationships. A route that forgets to filter manually still can't leak another clinic's data.
2. **Postgres RLS** ā€” migration `a3f9c2d81e47_add_row_level_security.py` (plus `f3f85dc00800_add_rls_to_subscription_payments.py`, `22a40bfa3b04_add_payment_plan_installments.py`, and `c3d67ef01a24_add_clinic_id_to_payments_invoice_items.py` for tables added/patched later) adds `FORCE ROW LEVEL SECURITY` policies on the same tables, keyed off the `app.current_clinic_id` / `app.bypass_rls` session GUCs. This is defense-in-depth in case a future code path uses raw SQL. The `clinic_isolation` policy expression wraps both `current_setting(...)` calls in `(select ...)` (fixed 2026-07-10 in `b7d924fa61ec`) ā€” calling it unwrapped makes Postgres re-evaluate it per row instead of caching it once per query (Supabase's Performance Advisor flags this as "Auth RLS Initialization Plan"); copy the wrapped form for any new policy.

**Whenever a new table gets a `clinic_id` column, it must be added to *both* lists** ā€” `_scoped_models()` in `tenancy.py` and a migration enabling RLS on it (copy the pattern in `c3d67ef01a24_...` or `22a40bfa3b04_...`, which do it correctly from the start instead of as a same-day patch). `subscription_payments` shipped without either for one day (2026-06-21) before this was caught and fixed; it was never actually exploitable (every route touching it was already gated by `platform_admin_required`, see "Platform administration" below), but don't rely on route-level gating alone ā€” that's exactly the kind of single point of failure this dual-layer design exists to avoid. Tables that are genuinely platform-wide config with no `clinic_id` (e.g. `subscription_tiers`, `clinics` itself, `pages`) correctly stay out of both lists ā€” they instead get RLS enabled with a plain `TO dental_app USING (true)` policy (no per-row isolation, just blocking Supabase's PostgREST anon/authenticated roles from reading them directly; see `f2a891cd47e0_add_rls_to_platform_config_tables.py`). `alembic_version` gets RLS enabled with **no policy at all** (`a1c56de89f03`) since the runtime role never has a legitimate reason to touch it ā€” don't add `FORCE` there, it would break `flask db upgrade` itself.

**Previously an open gap, fixed 2026-07-10** (`c3d67ef01a24_add_clinic_id_to_payments_invoice_items.py`): `payments` (individual `Payment` rows against an `Invoice`) and `invoice_items` both predate multi-tenancy and had no `clinic_id` column ā€” neither was in either enforcement layer. Closed by adding the column (backfilled from the parent `Invoice.clinic_id`), registering both in `_scoped_models()`, and adding the standard RLS policy. Every route that constructs a `Payment`/`InvoiceItem` now sets `clinic_id=invoice.clinic_id` explicitly (`routes/billing.py`) ā€” the ORM filter only scopes `SELECT`s, not inserts, so any new construction site must set `clinic_id` itself, same as every other scoped model.

`g.clinic_id` is resolved once per request in `resolve_request_clinic()` (registered as a Flask `before_request` hook in `app/__init__.py`) from the JWT identity. Key consequences for anyone touching this code:
- `is_platform_admin` users get an unscoped session (`g.clinic_id = None`, `g.rls_bypass = True`) ā€” intentional, for cross-clinic platform staff (see "Platform administration" below).
- Users with no resolvable `clinic_id` are filtered on `NO_MATCH_CLINIC_ID = -1` (fails closed, matches nothing) rather than skipping the filter.
- A rare platform-wide lookup (e.g. email-uniqueness at signup, in `seeder.py`'s `_email_taken`) must opt out of *both* layers explicitly: `.execution_options(skip_clinic_filter=True)` for the ORM filter, and the `platform_wide_lookup()` context manager (or `_bypass_rls()` in CLI scripts) for RLS.
- The app connects to Postgres as two different roles: `dental_app` (restricted, no `BYPASSRLS`, used for runtime traffic ā€” `DATABASE_URL`) and `dental_user` (owns the schema, used only for migrations ā€” `MIGRATIONS_DATABASE_URL`). This split is required because RLS is silently bypassed for a table's owning role. See `create_app_role.sql` for the role/grants setup and `entrypoint.sh` for which URL is used when.
- The `app.current_clinic_id` / `app.bypass_rls` GUCs are re-stamped on *every* SQLAlchemy connection-pool checkout (`@event.listens_for(Pool, "checkout")` in `tenancy.py`), reading `g.clinic_id`/`g.rls_bypass` fresh at that moment ā€” not set once per request via `db.session.commit()` like an earlier version did. A commit releases the connection back to the pool, so under real concurrency (gunicorn with multiple workers/threads) a later query in the same request could otherwise land on a different physical connection carrying another request's stale tenant context. There is no `teardown_request` GUC-reset hook anymore ā€” every checkout already stamps fresh state, so there's nothing to leak between requests.
- **`pool_pre_ping` is deliberately OFF** (fixed 2026-07-10, live in prod since 2026-07-11) ā€” the GUC-stamp query above already runs a round-trip on every checkout regardless, so it doubles as the pool's liveness check: `_stamp_tenant_guc_on_checkout` catches any failure and re-raises `sqlalchemy.exc.DisconnectionError`, which tells the pool to discard the connection and retry, exactly what `pool_pre_ping` would do ā€” just without paying for a separate `SELECT 1` round-trip first. Don't re-enable `pool_pre_ping` in `app/__init__.py`'s `SQLALCHEMY_ENGINE_OPTIONS` without removing this exception-handling first, or every checkout pays for both.

### RBAC: roles vs. per-clinic page permissions
There are two separate authorization mechanisms layered on top of each other:
- **Route-level role decorators** in `app/middleware/auth.py` (`admin_required`, `doctor_or_admin_required`, `clinical_access_required`, etc.) gate individual endpoints by `UserRole` (`admin`, `doctor`, `receptionist`, `assistant`, `guest`).
- **Page-level permissions** (`app/models/permission.py`: `Page` + `RolePermission`) let each clinic's admin customize which roles can view/create/edit/delete each app section, via the Permissions UI (`/permissions`, admin-only). `RolePermission` is clinic-scoped (each clinic gets its own matrix, seeded with sane defaults by `seed_pages()` in `app/utils/seeder.py`). The frontend's `roleGuard` (`frontend/src/app/core/guards/auth.guard.ts`) checks `route.data.pageKey` against `PermissionService.canView()`, which is populated from `GET /api/permissions/me` right after login (`AuthService.login()` chains into `PermissionService.load()`).

`GET /permissions/me` (`routes/permissions.py`) must build its `pages` response by filtering the already `sort_order`-ordered `Page.query` against the viewable-key set ā€” not by iterating the `RolePermission.query` result directly, which has no `order_by` and Postgres doesn't guarantee its row order. Doing the latter shipped a bug where non-admin roles (doctor, receptionist, assistant) saw the sidebar in an inconsistent order; admins never noticed because their `viewable` list is built straight from the ordered `Page` query (fixed 2026-07-05).

When adding a new app section: register it as a `Page` (typically via `STANDARD_PAGES` in `seeder.py` if it should exist for every clinic), add a route-level decorator on the backend endpoints, and add `data: { pageKey: '...' }` to the Angular route so `roleGuard` and the sidebar (which calls `accessiblePages()`) pick it up automatically. This `Page`/`RolePermission` system is **clinic-scoped configuration only** ā€” it lets a clinic's own admin customize their own staff's access. It has nothing to do with platform-admin access (next section); never gate a platform-admin endpoint with a `Page`/role check, and never let a clinic-scoped route depend on `is_platform_admin`.

### Platform administration: a separate SaaS-operator layer
Distinct from everything above, there's a second, non-clinic-scoped layer for the people who run the SaaS itself (onboarding clinics, tracking subscription payments, suspending non-payers) ā€” not to be mixed with clinic-facing code or the `Page`/`RolePermission` system, which is meaningless for staff who don't belong to any clinic:

- **`admin-frontend/`** is a wholly separate Angular app/codebase (own `package.json`, own routes, port 4300), not a section of `frontend/`. It has its own minimal `auth.service.ts` + `platform.service.ts` and its own `auth.guard.ts` ā€” it does not use `frontend/`'s `PermissionService`/`roleGuard`/page-permission machinery at all.
- Both frontends authenticate against the same `POST /api/auth/login` (a user is just a `User` row; `is_platform_admin` is a boolean flag on it, not a different login flow or token type) ā€” but every backend route `admin-frontend` calls lives in **`app/routes/platform_admin.py`** under `/api/platform/*`, gated by **`platform_admin_required`** (`app/middleware/auth.py`), which checks `user.is_platform_admin`. This is a different check from every other role decorator (`admin_required` etc.), which check `user.role` and are meaningless for platform staff (a platform admin's `role` is just `ADMIN` with no bearing on platform access; a clinic's own `ADMIN` must get a 403 from every `/api/platform/*` route).
- **Models**: `app/models/subscription.py` holds `SubscriptionTier` (a platform-wide plan/price list ā€” no `clinic_id`, not RLS/ORM-scoped, analogous to a config table) and `SubscriptionPayment` (one row per manually-recorded SaaS payment from a clinic ā€” has `clinic_id`, **is** RLS/ORM-scoped like any other tenant data, see "Multi-tenancy" above). `Clinic` itself carries the subscription state fields (`subscription_tier_id`, `subscription_status`, `trial_ends_at`, `next_payment_due_at`, `suspended_at`, `plan_started_at`, `plan_expires_at`) directly, since the clinic *is* the subscriber ā€” this is the one place tenant identity and billing state intentionally live together.
- Bootstrap a platform-admin user with `flask create-platform-admin` (`create_platform_admin()` in `seeder.py`) ā€” creates a `User` with `clinic_id=None`, `is_platform_admin=True`. `flask seed` also seeds default `SubscriptionTier` rows (`seed_subscription_tiers()`), including the `trial` tier (see below).
- Rule of thumb for future platform features: code for the SaaS-operator layer (models, routes, frontend) stays in its own files/app, gated only by `platform_admin_required`/`is_platform_admin`; it may *read* clinic data (platform admins are intentionally unscoped) but a clinic-facing route must never reach into platform-admin models or routes.

### SaaS subscription lifecycle & access control
Built 2026-06-22. A clinic's app access is gated by `Clinic.access_blocked()` (`app/models/clinic.py`), checked at two deliberately redundant layers ā€” same dual-enforcement philosophy as multi-tenancy above:
1. **`POST /api/auth/login`** (`routes/auth.py`) ā€” rejects with `403 {"error": "<friendly message>", "code": "clinic_access_blocked"}` before issuing any token. The primary, user-facing path.
2. **`resolve_request_clinic()`** (`middleware/tenancy.py`'s `before_request` hook, the same one that resolves `g.clinic_id`) ā€” re-checks on *every* authenticated request, for a token issued before the block took effect (e.g. trial expiring mid-session). Same response shape.

`access_blocked()` is `True` if: `Clinic.is_active` is `False` (platform-admin kill switch ā€” was previously unenforced despite the admin-frontend UI label "ClĆ­nica activa (acceso al sistema)" implying it already worked), `subscription_status` is `SUSPENDED`/`CANCELLED`, **or** `plan_expires_at` has passed (general-purpose, any status), **or** (legacy fallback) `subscription_status == TRIAL` and `trial_ends_at` has passed. `PAST_DUE` deliberately does *not* block ā€” it's a grace-period/dashboard-only signal (`overdue_clinics` in `GET /api/platform/dashboard`); nothing auto-transitions a clinic into it today.

`plan_started_at`/`plan_expires_at` are the CRUD-facing "fecha de inicio/vencimiento del plan" fields, editable via `PUT /api/platform/clinics/<id>` (admin-frontend's clinic-detail page has a view/edit toggle ā€” "Ver" lands read-only, "Editar" switches the same form to editable). They're what `access_blocked()` actually enforces, so extending `plan_expires_at` there genuinely un-blocks a clinic, it's not cosmetic. `record_payment()` keeps them in sync with whatever period was just paid (`period_start`/`period_end`), alongside the older `next_payment_due_at` (kept only for the dashboard's overdue stat).

New clinics (`create_clinic()` in `seeder.py`, used by both `flask create-clinic` and `POST /api/platform/clinics`) start on the **`trial`** `SubscriptionTier` ($0/mes, `max_users=10` ā€” same limits as `profesional`, seeded by `seed_subscription_tiers()`) for 30 days: `trial_ends_at`/`plan_expires_at` both set to `created_at + 30 days`. It's a real, distinct tier row rather than silently aliased to `profesional`, so admin-frontend's clinics list/detail clearly shows "Trial" instead of looking like the clinic already pays for Profesional. The create-clinic form can still override the tier explicitly.

`subscription_tier.max_users` is enforced in `routes/users.py` ā€” both `create_user()` and reactivating a user via `update_user()` (`is_active: true`) count *active* users against the assigned tier's limit (`None` = unlimited) and reject with `403` at capacity. Deactivated users don't count, so deactivating one frees a slot.

Frontend (`frontend/`, clinic-facing): `auth.interceptor.ts` detects `error.error?.code === 'clinic_access_blocked'` on *any* response, force-logs-out, and redirects to `/auth/login?blocked=<message>`; `login.component.ts` reads that query param into the same error banner used for normal login failures.

### Backend structure
- `app/models/` ā€” SQLAlchemy models, one file per domain (`clinic`, `subscription`, `user`, `patient`, `consultorio`, `appointment`, `appointment_type`, `treatment`, `billing`, `permission`). Every model exposes `to_dict()`; routes serialize through that rather than ad hoc dicts.
- `app/routes/` ā€” one blueprint per domain, registered in `app/__init__.py` under `/api/<name>` (e.g. `platform_admin.py` ā†’ `platform_bp` ā†’ `/api/platform`). Endpoints document themselves via Swagger YAML docstrings (parsed by `flasgger`); the aggregate spec is served at `/api/docs/` using the shared schemas/security scheme defined in `app/swagger_spec.py`.
- `app/middleware/` ā€” cross-cutting concerns only: `auth.py` (JWT + role decorators, including `platform_admin_required`) and `tenancy.py` (multi-tenancy, see above). Not a place for business logic.
- `app/utils/seeder.py` ā€” `flask seed` (demo-data for clinic #1 + default subscription tiers), `flask create-clinic` (onboard a real tenant: creates the `Clinic` row, default pages/permissions/appointment types, and first admin ā€” no demo data), and `flask create-platform-admin` (bootstrap a SaaS-operator user).
- `init_db.py` is a standalone script (not a blueprint), run directly with `python` inside the container; it has one-off `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` patches for columns added after the table already existed in some deployments ā€” prefer a real Alembic migration for new schema changes instead of extending that pattern.

### Frontend structure
This section is about `frontend/` (the clinic-facing app, port 4200). `admin-frontend/` (port 4300) is a separate, much smaller app ā€” see "Platform administration" above ā€” and does not follow any of this (no `Page`/permission system, no feature-per-backend-domain mirroring beyond `clinics`/`subscription-tiers`).
- `core/services/api.service.ts` holds one `@Injectable` service class per domain (`PatientService`, `AppointmentService`, `TreatmentService`, `BillingService`, `UserService`, `ConsultorioService`, `AppointmentTypeService`, `DashboardService`) ā€” all in this single file, thin wrappers around `HttpClient` calls to the matching Flask blueprint.
- `core/services/auth.service.ts` holds auth state in Angular signals (`currentUser`, `isLoggedIn`) backed by `localStorage`; role-check computed signals (`isAdmin`, `canManageBilling`, etc.) mirror the backend's role groupings and are used to conditionally show UI, but are not a substitute for backend authorization.
- `core/interceptors/auth.interceptor.ts` attaches the JWT bearer token and transparently retries once on a 401 via `AuthService.refreshToken()`.
- `app.routes.ts` lazy-loads each feature (`loadComponent`/`loadChildren`); every protected child route carries `canActivate: [roleGuard]` and `data: { pageKey, roles? }` ā€” `roles` is an extra hard gate on top of the page-permission check, used for sections that should never be configurable away from admin (e.g. `users`, `appointment-types`, `consultorios`).
- Feature modules under `features/` mirror the backend domains 1:1, plus a few frontend-only concerns: `calendar` (visual scheduling using `angular-calendar`), and `patients/odontogram.component.ts` + `medical-history.component.ts` (sub-views of a patient, not separate backend blueprints ā€” they hit `PatientService.getOdontogram/saveOdontogram` and the patient's `medical_history` JSON column). The calendar (`calendar.component.ts`) colors each event by doctor using `DOCTOR_COLORS`, with two special overrides: `cancelled`/`no_show` ā†’ `appt-inactive` CSS class (45% opacity, strikethrough title); `completed` ā†’ `COMPLETED_COLOR` gray (`#a0aec0`/`#edf2f7`) + `appt-completed` CSS class (60% opacity, italic title). Every event title includes the appointment status label as the last segment: `Paciente Ā· Dr. Apellido Ā· Consultorio Ā· Estado`.
- When a form's fields are needed inside a modal/dialog elsewhere (e.g. creating a patient from the appointment form), embed the existing feature component with an `embedded` input rather than duplicating its fields inline ā€” that's the established pattern in this codebase.
- `app.config.ts` enables `provideRouter(routes, withComponentInputBinding())`. **Gotcha**: for a component that's loaded directly as a route's `loadComponent` (not just used as a child in a parent template), the router calls `setInput()` for *every* declared `@Input()` on every navigation ā€” even ones with no matching route param/queryParam/data key, passing `undefined` and silently overwriting the field's own default/initializer. This only affects the component when it's the *routed* one; the same component used as an ordinary child elsewhere (a parent template binding `[someInput]="..."`) is unaffected. Bit `CalendarComponent` (`embedded`/`showHeader` inputs silently became `undefined` on the routed `/calendar` page, hiding its title and collapsing its layout, while the same component worked fine embedded in the dashboard) on 2026-06-22. Fix: declare such inputs as accessor pairs with a `??` fallback in the setter (`set x(v) { this._x = v ?? default; }`) instead of a plain `@Input() x = default;` field ā€” any other component that's *both* routed *and* embedded elsewhere needs the same treatment.

### Domain model notes
- `Appointment` can optionally link to a `TreatmentPlan` (multi-session treatments, with `session_number`) and to a `Consultorio` (room). `Treatment` and `Invoice` each have an optional 1:1 back-reference to the `Appointment` that produced them. `AppointmentStatus` has 6 values: `scheduled`, `confirmed`, `in_progress`, `completed`, `cancelled`, `no_show`. **`completed`, `cancelled`, and `no_show` all free up the doctor and consultorio** ā€” all three are excluded from availability conflict checks. Backend: `check_doctor_availability()` / `check_consultorio_availability()` in `routes/appointments.py` and the `/appointments/availability` endpoint each use `.not_in([CANCELLED, NO_SHOW, COMPLETED])`. Frontend: `appointment-form.component.ts` does **not** use the `/appointments/availability` endpoint for the doctor check ā€” both doctor and consultorio conflicts are detected via a single `getAll(all=true, per_page=200)` call filtered client-side (status exclusion list `['cancelled', 'no_show', 'completed']`). This means availability display works without a backend restart after code changes, and both checks are always consistent. `AppointmentService.checkAvailability()` remains in `api.service.ts` (the backend endpoint still exists) but is no longer called by the form.
- `Invoice.recalculate()` is the single source of truth for `subtotal`/`total`/`balance`/`status` ā€” call it after mutating `items`, `discount`, or `amount_paid` rather than computing those fields inline. **`InvoiceStatus.PARTIAL` is reinstated and active again** (2026-07-12) ā€” `POST /invoices/<id>/payments` accepts any `0 < amount <= balance` (the old "solo pagos totales" restriction was deliberately removed), and `recalculate()` sets `status = PARTIAL` whenever `0 < amount_paid` and `balance > 0`, `PAID` when `balance == 0`, `PENDING` otherwise. The frontend has a "Pago Completo"/"Pago Parcial" toggle on the payment panel (`invoice-detail.component`); `invStatusLabel()`/`.inv-partial` handle the new status everywhere it's rendered (`invoice-detail`, `billing.component`'s list + `statusFilters`). `recalculate()` leaves `CANCELLED` alone (returns early) instead of deriving status from balance ā€” until 2026-06-22 it didn't, so setting `status=cancelled` via `PUT /invoices/<id>` got silently reverted back to `pending`/`paid` by the very next `recalculate()` call in the same request; if you touch this method again, re-verify a cancel actually sticks by `GET`-ing the invoice back afterward, not just trusting the response of the call that cancelled it.
- An invoice can only be cancelled (`PUT /invoices/<id>` with `status: cancelled`) or have its `items` edited while `status == pending` ā€” both are blocked once any payment (partial or full) has been registered, since that flips `status` away from `pending`. Cancelled invoices are excluded from `total_invoiced`/`total_collected` in `GET /billing/summary` and from `monthly_revenue` in the main dashboard; both of those already summed by amount fields with only a `!= CANCELLED` filter (not an exact `status` match), so they required **no changes** for the `PARTIAL` reinstatement ā€” the aggregates were already correct for partially-paid invoices. `GET /billing/summary`'s `pending_balance` was the one aggregate that DID need a fix (2026-07-12) ā€” it filtered `status == PENDING` only, silently excluding partially-paid invoices' outstanding balance from the Cobros summary card; now `status.in_([PENDING, PARTIAL])`, matching the dashboard's `monthly_pending` (`routes/dashboard.py`) which already had this right. If you add another balance aggregate, default to the `.in_([PENDING, PARTIAL])` form, not a single-status filter. The frontend's invoice-detail page has a "Cancelar Factura" button (top-right, pending only) and an "Editar" toggle on the items table (pending only) that switches it to the same item-row form as the create-invoice page.
- `PaymentMethod` only offers `cash`/`qr` for new payments (`ALLOWED_PAYMENT_METHODS` in `routes/billing.py`) ā€” `card`/`transfer`/`other` are legacy values kept solely so old `Payment`/simulated-data rows still load; don't resurrect them in any new UI.
- **`PaymentPlan` cuotas are fixed cost-per-cita** (redesigned FCLI-13, 2026-07-12). `installments` = fixed count of citas, `installment_amount` = fixed cost per cita; both are set at creation via `calc_mode` (`'per_cita'`: the caller sends `cost_per_cita` and the server derives `total_amount = down_payment + num_citas*cost_per_cita`; `'total'`: the caller sends `total_amount` and the server derives `cost_per_cita`) and never recomputed by registering a payment. `installment_amount` itself is fixed, but **the amount actually paid per registration is variable again** (corrected same day, 2026-07-12) ā€” `POST /payment-plans/<id>/installment` takes either `{count}` ("pago completo": N whole citas at once, server computes `amount = count * installment_amount`) or `{amount}` ("pago parcial": any amount, can be less than one cita) ā€” both validated only against `0 < amount <= balance`, no multiple-of-`installment_amount` requirement. `paid_installments` is **derived**, not incremented directly: `min(installments, floor((total_paid - down_payment) / installment_amount))` ā€” the count of *fully* covered citas. `PaymentPlan.partial_progress_amount` (a computed property, exposed in `to_dict()`) is the leftover credit toward the next not-yet-fully-covered cita, used by the frontend to render a third "Parcial" state (with amount) on that cita row, alongside "Pagada"/"Pendiente". Completion (`status ā†’ completed`) is driven by `balance <= 0` (equivalent to `total_paid >= total_amount`), with `paid_installments >= installments` kept as a secondary OR condition. The guardrail on `PUT /payment-plans/<id>` that locks `down_payment`/`total_amount`/`cost_per_cita`/`calc_mode` once real cita money has been collected checks **`total_paid > down_payment`**, not `paid_installments > 0` ā€” a partial payment can leave `paid_installments` at 0 while still having collected real money toward cita 1, so the paid-installments-based check would have let the cost be edited out from under an already-partially-paid cita. `num_citas` may still increase at any time (never decrease below `paid_installments`), plus `name`/`notes`/`start_date`/`end_date` always. `PaymentPlan.end_date` is a class-2 calendar date like `start_date` ā€” bare `.isoformat()`/`formatDateOnly()`, never `iso_utc()`/`new Date()`. **A new plan is created with `total_paid = 0` ā€” the `down_payment`/enganche is NOT auto-marked as paid at creation** (changed 2026-07-12; it previously initialized `total_paid = down_payment` and inserted a "Pago inicial / enganche" `PaymentPlanInstallment` row on creation). The plan is created "enabled" with the whole `total_amount` (including the enganche) as outstanding `balance`; the enganche is then collected like any other payment via `POST /payment-plans/<id>/installment` (typically a "pago parcial" of the `down_payment` amount). The `paid_installments`/`partial_progress_amount` derivation subtracts `down_payment` from `total_paid`, so the first `down_payment` of money collected is attributed to the enganche and only money beyond that advances the citas ā€” the enganche row on the detail/print pages (`payment-plan-detail.component`/`payment-plan-print.component`, `buildCitasRows()`) is rendered `Pagada`/`Parcial`/`Pendiente` by comparing `total_paid` against `down_payment` (NOT hardcoded to `paid` anymore). Each registered payment is logged as its own `PaymentPlanInstallment` row (`payment_plan_installments` table) ā€” `GET /payment-plans/<id>/installments` returns that history, most recent first. Each row also stores a **snapshot** of the plan's cumulative state right after that payment (`total_paid_after`/`balance_after`, added 2026-07-12 in `d8e1f4a7b920`) so a printed payment receipt (`payment-receipt-print.component`, route `billing/payment-plans/:id/comprobante/:installmentId/imprimir`) shows the total-paid/balance AS OF that payment, not the plan's live values ā€” set at write time in `register_installment` from `plan.total_paid`/`plan.balance` (which are already post-payment at that point), and backfilled for pre-existing rows by replaying the ordered ledger. The receipt derives the historical plan total as `total_paid_after + balance_after`; both columns are nullable and the frontend falls back to live plan values only for any legacy row left null. Plans created before this table existed (2026-06-22) have no per-payment rows for their prior progress; that endpoint detects the gap (`total_paid` > sum of logged rows) and prepends a synthetic "Pagos registrados antes de este historial" entry for the difference so the list always sums to `total_paid` ā€” don't try to backfill real rows for that gap. Plans created before the FCLI-13 redesign keep whatever `installment_amount`/`paid_installments` they already had (no data migration) and are rendered the same way. The billing list page's plan cards link to `Ver`/`Editar` (`/billing/payment-plans/:id` and `.../edit`) rather than embedding any payment-registration UI inline on the card itself ā€” registering a payment (a "Pago Completo"/"Pago Parcial" toggle, same visual pattern as the Comprobantes payment panel above) happens on the detail page, not the card. The detail page also has an "Imprimir" button opening `/billing/payment-plans/:id/imprimir` (`payment-plan-print.component`, outside the `LayoutComponent` shell, same printable-document pattern as receta/historia mĆ©dica below) in a new tab via `window.open`.
- **Shared "condiciones" fields** (`frontend/src/app/shared/components/billing-conditions-fields/`, built for FCLI-13, reused as-is by FCLI-14/Presupuestos below): a presentational component taking `@Input() form!: FormGroup` ā€” the parent must build a nested `FormGroup` with EXACTLY these controls and pass that instance down (no `ControlValueAccessor`, just a shared `FormGroup` reference across the component boundary): `calc_mode` (`'per_cita'|'total'`), `num_citas`, `cost_per_cita`, `total_amount`, `down_payment`, `start_date`, `end_date`. It owns the live calc-mode derivation (recomputing whichever of `cost_per_cita`/`total_amount` isn't the one the user is editing) and exposes three read-only-ish inputs: `@Input() costLocked` renders all cost fields read-only once a plan/presupuesto already has payments; `@Input() lockToTotal` (added 2026-07-12) disables the calc-mode toggle and forces the 'total' flow ā€” currently set `true` by **both** `payment-plan-form` and `budget-form` so only "Por monto total" is offered for now (parents must also set the form's `calc_mode` to `'total'`; `isPerCita()` treats `lockToTotal` as authoritative so the display never gets stuck in per-cita mode); `@Input() totalReadonly` (added 2026-07-12, budget-only) renders just the Monto Total field read-only when the total is driven externally. `payment-plan-form.component` and `budget-form.component` both embed it as `conditions: fb.group({...})` and read `form.get('conditions')` ā€” don't duplicate these fields in any future form that needs a citas/cost schedule.

### Timestamps & timezones
Fixed 2026-07-10. Datetime fields fall into four classes ā€” **don't lump them together**, that's exactly the bug that originally shipped (instants displaying hours off, date-only fields displaying a day early, both from calling `new Date(isoString)` on backend output with no UTC marker):
1. **Instant** (`db.DateTime`, `default=utcnow` ā€” `created_at`, `performed_at`, `payment_date`, etc.): real value is UTC. Backend serializes with `app/utils/serialization.py`'s `iso_utc()` (appends a literal `Z`) instead of bare `.isoformat()`. Frontend then just does `new Date(iso)` ā€” correct once the `Z` is present.
2. **Calendar date** (`db.Date`, no time component ā€” `date_of_birth`, `due_date`, `start_date`, etc.): backend serializes as bare `YYYY-MM-DD` (unchanged). Frontend must NOT do `new Date(iso)` on these (parses as UTC midnight, displays a day early in any UTC-negative timezone) ā€” use `formatDateOnly()` from `core/util/date.util.ts` (own copy in each frontend app), which parses the `Y-M-D` parts directly instead of going through `Date`'s ISO parser.
3. **Naive-midnight date-picker field** (`Clinic.plan_started_at`/`plan_expires_at`): already displays correctly as-is (naive `new Date()` parses as local midnight = same day) ā€” deliberately left untouched; adding `Z` here would shift it a day.
4. **Local wall-clock** (`Appointment.scheduled_at`): naive-local end to end, backed by `clinic_time.py`'s `local_now()`/`local_today()` on the backend side for comparisons ā€” also left untouched.

When adding a new datetime column, classify it into one of these four before writing `to_dict()` ā€” most new fields are class 1 (`iso_utc()`) or class 2 (`formatDateOnly()` on the frontend, plain `.isoformat()` on the backend). `date.util.ts`'s `formatDate`/`formatDateTime`/`formatTime` are for classes 1 and 4 only ā€” never use them on a class-2 field.
