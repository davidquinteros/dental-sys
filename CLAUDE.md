# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

DentalSys — a multi-tenant SaaS MVP for managing a dental clinic (patients, appointments, clinical treatments, billing, and a calendar/odontogram). Backend: Flask (Python) + PostgreSQL. Two separate Angular 18 frontends share that one backend: `frontend/` (port 4200, used by clinic staff — patients/appointments/billing/etc., gated by the RBAC described below) and `admin-frontend/` (port 4300, used only by the SaaS operator to manage clinics/subscriptions — see "Platform administration" below). UI strings, route labels, and seed data are in Spanish.

The root `README.md` (rewritten 2026-07-11 to match current multi-tenant architecture) is the general project overview; this file is the deeper, gotcha-level guide for anyone (human or agent) working on the code — keep both in sync when a change touches architecture, not just this file.

## Working agreement

**Never `git commit` or `git push` automatically.** All commits and pushes are done manually by the user — staging/committing/pushing is never an implicit step of finishing a task, even after a working fix, even right before reporting completion. Prepare and describe changes; wait to be explicitly asked before committing or pushing.

## Commands

### Docker (primary workflow)
```bash
docker compose up -d --build       # starts backend (:5000), frontend (:4200), admin-frontend (:4300) — no local db container; points at a shared Supabase project instead, see "Deployment" below
docker compose logs -f backend     # tail backend logs
docker compose exec backend flask db migrate -m "message"   # create a migration
docker compose exec backend flask db upgrade                # apply migrations
docker compose exec backend flask seed                      # seed clinic #1 demo data + default subscription tiers
docker compose exec backend flask create-clinic --name "..." --admin-email "..." --admin-password "..."   # defaults to the "Trial" tier, 30-day plan_expires_at — see "SaaS subscription lifecycle" below
docker compose exec backend flask create-platform-admin --email "..." --password "..."   # bootstrap a SaaS-operator user (is_platform_admin=True, no clinic)
```
`backend/entrypoint.sh` runs migrations (as the `dental_user` superuser-ish role, via `MIGRATIONS_DATABASE_URL`) then execs `gunicorn` (as the restricted `dental_app` role, via `DATABASE_URL`) automatically on container start — see "Multi-tenancy" below for why two DB roles exist. **It does NOT run `flask seed`** (removed 2026-06-22) — seeding/demo data is always a deliberate, manual step (`docker compose exec backend flask seed`), never triggered by a container start/restart/crash-recovery, so a restart can never silently inject demo users/patients into real tenant data. Gunicorn has no `--reload` flag here (tuned to behave like prod even locally), so it loads every backend module once at boot: any `.py` change (routes, models, middleware) needs `docker compose restart backend` to actually take effect, even though the file is bind-mounted and visible inside the container immediately.

### Backend (Flask), run outside Docker
```bash
cd backend
python3 -m venv venv && source venv/bin/activate   # venv\Scripts\activate on Windows
pip install -r requirements.txt
cp .env.example .env        # then edit DATABASE_URL etc.
flask db upgrade
flask seed
flask run --port 5000
```
There is no automated test suite (no pytest files) and no linter configured for the backend — verify changes by exercising the API (e.g. via `/api/docs/`, the Swagger UI) or `flask shell`.

### Frontend (Angular), run outside Docker
```bash
cd frontend
npm install
npm start          # ng serve --port 4200
npm run build       # ng build
npm test            # karma/jasmine — no spec files currently exist in the repo
```
No ESLint config is present. `admin-frontend/` is a separate Angular app/`package.json` with the same commands, served on port 4300 (`npm start` there runs `ng serve --port 4300`).

**Live UI verification when there's no host-side browser available**: install a real browser inside the already-running `frontend` container and drive it with Puppeteer rather than guessing from source — `docker compose exec frontend sh -c "apk add --no-cache chromium && npm install --no-save puppeteer-core"`, then launch with `puppeteer.launch({ executablePath: '/usr/bin/chromium-browser', headless: 'new', args: ['--no-sandbox', '--disable-setuid-sandbox', '--host-resolver-rules=MAP localhost:5000 backend:5000'] })`. The `--host-resolver-rules` flag matters: `environment.ts`'s `apiUrl` is `http://localhost:5000`, which only resolves correctly in a real browser on the host — from *inside* the frontend container, `localhost:5000` is the frontend container itself, so API calls get `ERR_CONNECTION_REFUSED` without that flag redirecting them to the `backend` service. `window.ng.getComponent(domElement)` (Angular's dev-mode debug API, available since `environment.ts` has `production: false`) is useful for inspecting a component instance's actual runtime property values directly — caught a real bug (see the `withComponentInputBinding()` gotcha above) that wasn't visible from the DOM or source alone.

### Single migration / one-off DB script
Migrations live in `backend/migrations/versions/`; create with `flask db migrate -m "..."` then hand-edit if Alembic's autogenerate misses something non-trivial (e.g. enabling RLS, see `a3f9c2d81e47_add_row_level_security.py`, is written by hand with raw `op.execute`).

## Deployment

Two independent environments, each pairing a Render backend + two Render Static Site frontends with its own separate Supabase Postgres project (fully separate databases and Storage buckets, not just config-separated):
- **Production** — tracks `main`. Backend `dental-sys-backend` runs on Render's native Python runtime, NOT Docker (`backend/Dockerfile`/`entrypoint.sh` are never used there — the Start Command is plain `gunicorn ... run:app`, configured directly in Render's dashboard).
- **Testing** — tracks `testing`. Same shape (`dental-sys-backend-testing` + two static sites), a separate Supabase project. `frontend/` and `admin-frontend/` each have a `testing` Angular build configuration + `environment.testing.ts` pointing at the testing backend's URL.

**Local dev shares the testing database.** `docker compose up` has no local Postgres container; `DATABASE_URL`/`MIGRATIONS_DATABASE_URL` in the root `.env` (gitignored, separate from `backend/.env`) point at the testing Supabase project's session pooler. Running `flask seed` or any destructive local action affects the same data the `testing` Render environment serves — there is no disposable local-only database to fall back on.

**Render never runs migrations automatically**, for either environment — no `preDeployCommand`, and prod's native runtime doesn't execute `entrypoint.sh` at all. Every schema change needs a manual `flask db upgrade` against the target environment's schema-owner role, run before/alongside deploying the code that depends on it; prod has drifted behind head this way more than once. When running `flask db` commands manually (outside `entrypoint.sh`'s automatic override), always reproduce its role override explicitly — `DATABASE_URL="$MIGRATIONS_DATABASE_URL" flask db upgrade` locally, or the equivalent prod URL from `backend/.env` for prod — running it bare uses the restricted runtime role, which lacks DDL rights and, on the testing Supabase project specifically, can't even read its own `alembic_version` row (that project auto-enables RLS with no policy on every new table it creates — see the multi-tenancy RLS notes above for the general pattern this falls under). Both Supabase projects use the same two-role split as local (`create_app_role.sql`): a restricted `dental_app` role for runtime, and the schema-owning `postgres` role for migrations only.

## Architecture

### Multi-tenancy: two independent enforcement layers
Every clinic-scoped table (`users`, `patients`, `appointments`, `treatments`, `treatment_plans`, `invoices`, `invoice_items`, `payments`, `payment_plans`, `payment_plan_installments`, `consultorios`, `appointment_types`, `role_permissions`, `subscription_payments`, `treatment_images`) has a `clinic_id`. Isolation between clinics is enforced twice, deliberately:

1. **ORM-level filter** — `backend/app/middleware/tenancy.py`'s `_scoped_models()` list registers a SQLAlchemy `do_orm_execute` session event that injects `clinic_id == g.clinic_id` into every SELECT against those models, including ones triggered by lazy-loaded relationships. A route that forgets to filter manually still can't leak another clinic's data.
2. **Postgres RLS** — migration `a3f9c2d81e47_add_row_level_security.py` (plus `f3f85dc00800_add_rls_to_subscription_payments.py`, `22a40bfa3b04_add_payment_plan_installments.py`, and `c3d67ef01a24_add_clinic_id_to_payments_invoice_items.py` for tables added/patched later) adds `FORCE ROW LEVEL SECURITY` policies on the same tables, keyed off the `app.current_clinic_id` / `app.bypass_rls` session GUCs. This is defense-in-depth in case a future code path uses raw SQL. The `clinic_isolation` policy expression wraps both `current_setting(...)` calls in `(select ...)` (fixed 2026-07-10 in `b7d924fa61ec`) — calling it unwrapped makes Postgres re-evaluate it per row instead of caching it once per query (Supabase's Performance Advisor flags this as "Auth RLS Initialization Plan"); copy the wrapped form for any new policy.

**Whenever a new table gets a `clinic_id` column, it must be added to *both* lists** — `_scoped_models()` in `tenancy.py` and a migration enabling RLS on it (copy the pattern in `c3d67ef01a24_...` or `22a40bfa3b04_...`, which do it correctly from the start instead of as a same-day patch). `subscription_payments` shipped without either for one day (2026-06-21) before this was caught and fixed; it was never actually exploitable (every route touching it was already gated by `platform_admin_required`, see "Platform administration" below), but don't rely on route-level gating alone — that's exactly the kind of single point of failure this dual-layer design exists to avoid. Tables that are genuinely platform-wide config with no `clinic_id` (e.g. `subscription_tiers`, `clinics` itself, `pages`) correctly stay out of both lists — they instead get RLS enabled with a plain `TO dental_app USING (true)` policy (no per-row isolation, just blocking Supabase's PostgREST anon/authenticated roles from reading them directly; see `f2a891cd47e0_add_rls_to_platform_config_tables.py`). `alembic_version` gets RLS enabled with **no policy at all** (`a1c56de89f03`) since the runtime role never has a legitimate reason to touch it — don't add `FORCE` there, it would break `flask db upgrade` itself.

**Previously an open gap, fixed 2026-07-10** (`c3d67ef01a24_add_clinic_id_to_payments_invoice_items.py`): `payments` (individual `Payment` rows against an `Invoice`) and `invoice_items` both predate multi-tenancy and had no `clinic_id` column — neither was in either enforcement layer. Closed by adding the column (backfilled from the parent `Invoice.clinic_id`), registering both in `_scoped_models()`, and adding the standard RLS policy. Every route that constructs a `Payment`/`InvoiceItem` now sets `clinic_id=invoice.clinic_id` explicitly (`routes/billing.py`) — the ORM filter only scopes `SELECT`s, not inserts, so any new construction site must set `clinic_id` itself, same as every other scoped model.

`g.clinic_id` is resolved once per request in `resolve_request_clinic()` (registered as a Flask `before_request` hook in `app/__init__.py`) from the JWT identity. Key consequences for anyone touching this code:
- `is_platform_admin` users get an unscoped session (`g.clinic_id = None`, `g.rls_bypass = True`) — intentional, for cross-clinic platform staff (see "Platform administration" below).
- Users with no resolvable `clinic_id` are filtered on `NO_MATCH_CLINIC_ID = -1` (fails closed, matches nothing) rather than skipping the filter.
- A rare platform-wide lookup (e.g. email-uniqueness at signup, in `seeder.py`'s `_email_taken`) must opt out of *both* layers explicitly: `.execution_options(skip_clinic_filter=True)` for the ORM filter, and the `platform_wide_lookup()` context manager (or `_bypass_rls()` in CLI scripts) for RLS.
- The app connects to Postgres as two different roles: `dental_app` (restricted, no `BYPASSRLS`, used for runtime traffic — `DATABASE_URL`) and `dental_user` (owns the schema, used only for migrations — `MIGRATIONS_DATABASE_URL`). This split is required because RLS is silently bypassed for a table's owning role. See `create_app_role.sql` for the role/grants setup and `entrypoint.sh` for which URL is used when.
- The `app.current_clinic_id` / `app.bypass_rls` GUCs are re-stamped on *every* SQLAlchemy connection-pool checkout (`@event.listens_for(Pool, "checkout")` in `tenancy.py`), reading `g.clinic_id`/`g.rls_bypass` fresh at that moment — not set once per request via `db.session.commit()` like an earlier version did. A commit releases the connection back to the pool, so under real concurrency (gunicorn with multiple workers/threads) a later query in the same request could otherwise land on a different physical connection carrying another request's stale tenant context. There is no `teardown_request` GUC-reset hook anymore — every checkout already stamps fresh state, so there's nothing to leak between requests.

### RBAC: roles vs. per-clinic page permissions
There are two separate authorization mechanisms layered on top of each other:
- **Route-level role decorators** in `app/middleware/auth.py` (`admin_required`, `doctor_or_admin_required`, `clinical_access_required`, etc.) gate individual endpoints by `UserRole` (`admin`, `doctor`, `receptionist`, `assistant`, `guest`).
- **Page-level permissions** (`app/models/permission.py`: `Page` + `RolePermission`) let each clinic's admin customize which roles can view/create/edit/delete each app section, via the Permissions UI (`/permissions`, admin-only). `RolePermission` is clinic-scoped (each clinic gets its own matrix, seeded with sane defaults by `seed_pages()` in `app/utils/seeder.py`). The frontend's `roleGuard` (`frontend/src/app/core/guards/auth.guard.ts`) checks `route.data.pageKey` against `PermissionService.canView()`, which is populated from `GET /api/permissions/me` right after login (`AuthService.login()` chains into `PermissionService.load()`).

`GET /permissions/me` (`routes/permissions.py`) must build its `pages` response by filtering the already `sort_order`-ordered `Page.query` against the viewable-key set — not by iterating the `RolePermission.query` result directly, which has no `order_by` and Postgres doesn't guarantee its row order. Doing the latter shipped a bug where non-admin roles (doctor, receptionist, assistant) saw the sidebar in an inconsistent order; admins never noticed because their `viewable` list is built straight from the ordered `Page` query (fixed 2026-07-05).

When adding a new app section: register it as a `Page` (typically via `STANDARD_PAGES` in `seeder.py` if it should exist for every clinic), add a route-level decorator on the backend endpoints, and add `data: { pageKey: '...' }` to the Angular route so `roleGuard` and the sidebar (which calls `accessiblePages()`) pick it up automatically. This `Page`/`RolePermission` system is **clinic-scoped configuration only** — it lets a clinic's own admin customize their own staff's access. It has nothing to do with platform-admin access (next section); never gate a platform-admin endpoint with a `Page`/role check, and never let a clinic-scoped route depend on `is_platform_admin`.

### Platform administration: a separate SaaS-operator layer
Distinct from everything above, there's a second, non-clinic-scoped layer for the people who run the SaaS itself (onboarding clinics, tracking subscription payments, suspending non-payers) — not to be mixed with clinic-facing code or the `Page`/`RolePermission` system, which is meaningless for staff who don't belong to any clinic:

- **`admin-frontend/`** is a wholly separate Angular app/codebase (own `package.json`, own routes, port 4300), not a section of `frontend/`. It has its own minimal `auth.service.ts` + `platform.service.ts` and its own `auth.guard.ts` — it does not use `frontend/`'s `PermissionService`/`roleGuard`/page-permission machinery at all.
- Both frontends authenticate against the same `POST /api/auth/login` (a user is just a `User` row; `is_platform_admin` is a boolean flag on it, not a different login flow or token type) — but every backend route `admin-frontend` calls lives in **`app/routes/platform_admin.py`** under `/api/platform/*`, gated by **`platform_admin_required`** (`app/middleware/auth.py`), which checks `user.is_platform_admin`. This is a different check from every other role decorator (`admin_required` etc.), which check `user.role` and are meaningless for platform staff (a platform admin's `role` is just `ADMIN` with no bearing on platform access; a clinic's own `ADMIN` must get a 403 from every `/api/platform/*` route).
- **Models**: `app/models/subscription.py` holds `SubscriptionTier` (a platform-wide plan/price list — no `clinic_id`, not RLS/ORM-scoped, analogous to a config table) and `SubscriptionPayment` (one row per manually-recorded SaaS payment from a clinic — has `clinic_id`, **is** RLS/ORM-scoped like any other tenant data, see "Multi-tenancy" above). `Clinic` itself carries the subscription state fields (`subscription_tier_id`, `subscription_status`, `trial_ends_at`, `next_payment_due_at`, `suspended_at`, `plan_started_at`, `plan_expires_at`) directly, since the clinic *is* the subscriber — this is the one place tenant identity and billing state intentionally live together.
- Bootstrap a platform-admin user with `flask create-platform-admin` (`create_platform_admin()` in `seeder.py`) — creates a `User` with `clinic_id=None`, `is_platform_admin=True`. `flask seed` also seeds default `SubscriptionTier` rows (`seed_subscription_tiers()`), including the `trial` tier (see below).
- Rule of thumb for future platform features: code for the SaaS-operator layer (models, routes, frontend) stays in its own files/app, gated only by `platform_admin_required`/`is_platform_admin`; it may *read* clinic data (platform admins are intentionally unscoped) but a clinic-facing route must never reach into platform-admin models or routes.

### SaaS subscription lifecycle & access control
Built 2026-06-22. A clinic's app access is gated by `Clinic.access_blocked()` (`app/models/clinic.py`), checked at two deliberately redundant layers — same dual-enforcement philosophy as multi-tenancy above:
1. **`POST /api/auth/login`** (`routes/auth.py`) — rejects with `403 {"error": "<friendly message>", "code": "clinic_access_blocked"}` before issuing any token. The primary, user-facing path.
2. **`resolve_request_clinic()`** (`middleware/tenancy.py`'s `before_request` hook, the same one that resolves `g.clinic_id`) — re-checks on *every* authenticated request, for a token issued before the block took effect (e.g. trial expiring mid-session). Same response shape.

`access_blocked()` is `True` if: `Clinic.is_active` is `False` (platform-admin kill switch — was previously unenforced despite the admin-frontend UI label "Clínica activa (acceso al sistema)" implying it already worked), `subscription_status` is `SUSPENDED`/`CANCELLED`, **or** `plan_expires_at` has passed (general-purpose, any status), **or** (legacy fallback) `subscription_status == TRIAL` and `trial_ends_at` has passed. `PAST_DUE` deliberately does *not* block — it's a grace-period/dashboard-only signal (`overdue_clinics` in `GET /api/platform/dashboard`); nothing auto-transitions a clinic into it today.

`plan_started_at`/`plan_expires_at` are the CRUD-facing "fecha de inicio/vencimiento del plan" fields, editable via `PUT /api/platform/clinics/<id>` (admin-frontend's clinic-detail page has a view/edit toggle — "Ver" lands read-only, "Editar" switches the same form to editable). They're what `access_blocked()` actually enforces, so extending `plan_expires_at` there genuinely un-blocks a clinic, it's not cosmetic. `record_payment()` keeps them in sync with whatever period was just paid (`period_start`/`period_end`), alongside the older `next_payment_due_at` (kept only for the dashboard's overdue stat).

New clinics (`create_clinic()` in `seeder.py`, used by both `flask create-clinic` and `POST /api/platform/clinics`) start on the **`trial`** `SubscriptionTier` ($0/mes, `max_users=10` — same limits as `profesional`, seeded by `seed_subscription_tiers()`) for 30 days: `trial_ends_at`/`plan_expires_at` both set to `created_at + 30 days`. It's a real, distinct tier row rather than silently aliased to `profesional`, so admin-frontend's clinics list/detail clearly shows "Trial" instead of looking like the clinic already pays for Profesional. The create-clinic form can still override the tier explicitly.

`subscription_tier.max_users` is enforced in `routes/users.py` — both `create_user()` and reactivating a user via `update_user()` (`is_active: true`) count *active* users against the assigned tier's limit (`None` = unlimited) and reject with `403` at capacity. Deactivated users don't count, so deactivating one frees a slot.

Frontend (`frontend/`, clinic-facing): `auth.interceptor.ts` detects `error.error?.code === 'clinic_access_blocked'` on *any* response, force-logs-out, and redirects to `/auth/login?blocked=<message>`; `login.component.ts` reads that query param into the same error banner used for normal login failures.

### Backend structure
- `app/models/` — SQLAlchemy models, one file per domain (`clinic`, `subscription`, `user`, `patient`, `consultorio`, `appointment`, `appointment_type`, `treatment`, `billing`, `permission`). Every model exposes `to_dict()`; routes serialize through that rather than ad hoc dicts.
- `app/routes/` — one blueprint per domain, registered in `app/__init__.py` under `/api/<name>` (e.g. `platform_admin.py` → `platform_bp` → `/api/platform`). Endpoints document themselves via Swagger YAML docstrings (parsed by `flasgger`); the aggregate spec is served at `/api/docs/` using the shared schemas/security scheme defined in `app/swagger_spec.py`.
- `app/middleware/` — cross-cutting concerns only: `auth.py` (JWT + role decorators, including `platform_admin_required`) and `tenancy.py` (multi-tenancy, see above). Not a place for business logic.
- `app/utils/seeder.py` — `flask seed` (demo-data for clinic #1 + default subscription tiers), `flask create-clinic` (onboard a real tenant: creates the `Clinic` row, default pages/permissions/appointment types, and first admin — no demo data), and `flask create-platform-admin` (bootstrap a SaaS-operator user).
- `init_db.py` is a standalone script (not a blueprint), run directly with `python` inside the container; it has one-off `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` patches for columns added after the table already existed in some deployments — prefer a real Alembic migration for new schema changes instead of extending that pattern.

### Frontend structure
This section is about `frontend/` (the clinic-facing app, port 4200). `admin-frontend/` (port 4300) is a separate, much smaller app — see "Platform administration" above — and does not follow any of this (no `Page`/permission system, no feature-per-backend-domain mirroring beyond `clinics`/`subscription-tiers`).
- `core/services/api.service.ts` holds one `@Injectable` service class per domain (`PatientService`, `AppointmentService`, `TreatmentService`, `BillingService`, `UserService`, `ConsultorioService`, `AppointmentTypeService`, `DashboardService`) — all in this single file, thin wrappers around `HttpClient` calls to the matching Flask blueprint.
- `core/services/auth.service.ts` holds auth state in Angular signals (`currentUser`, `isLoggedIn`) backed by `localStorage`; role-check computed signals (`isAdmin`, `canManageBilling`, etc.) mirror the backend's role groupings and are used to conditionally show UI, but are not a substitute for backend authorization.
- `core/interceptors/auth.interceptor.ts` attaches the JWT bearer token and transparently retries once on a 401 via `AuthService.refreshToken()`.
- `app.routes.ts` lazy-loads each feature (`loadComponent`/`loadChildren`); every protected child route carries `canActivate: [roleGuard]` and `data: { pageKey, roles? }` — `roles` is an extra hard gate on top of the page-permission check, used for sections that should never be configurable away from admin (e.g. `users`, `appointment-types`, `consultorios`).
- Feature modules under `features/` mirror the backend domains 1:1, plus a few frontend-only concerns: `calendar` (visual scheduling using `angular-calendar`), and `patients/odontogram.component.ts` + `medical-history.component.ts` (sub-views of a patient, not separate backend blueprints — they hit `PatientService.getOdontogram/saveOdontogram` and the patient's `medical_history` JSON column). The calendar (`calendar.component.ts`) colors each event by doctor using `DOCTOR_COLORS`, with two special overrides: `cancelled`/`no_show` → `appt-inactive` CSS class (45% opacity, strikethrough title); `completed` → `COMPLETED_COLOR` gray (`#a0aec0`/`#edf2f7`) + `appt-completed` CSS class (60% opacity, italic title). Every event title includes the appointment status label as the last segment: `Paciente · Dr. Apellido · Consultorio · Estado`.
- When a form's fields are needed inside a modal/dialog elsewhere (e.g. creating a patient from the appointment form), embed the existing feature component with an `embedded` input rather than duplicating its fields inline — that's the established pattern in this codebase.
- `app.config.ts` enables `provideRouter(routes, withComponentInputBinding())`. **Gotcha**: for a component that's loaded directly as a route's `loadComponent` (not just used as a child in a parent template), the router calls `setInput()` for *every* declared `@Input()` on every navigation — even ones with no matching route param/queryParam/data key, passing `undefined` and silently overwriting the field's own default/initializer. This only affects the component when it's the *routed* one; the same component used as an ordinary child elsewhere (a parent template binding `[someInput]="..."`) is unaffected. Bit `CalendarComponent` (`embedded`/`showHeader` inputs silently became `undefined` on the routed `/calendar` page, hiding its title and collapsing its layout, while the same component worked fine embedded in the dashboard) on 2026-06-22. Fix: declare such inputs as accessor pairs with a `??` fallback in the setter (`set x(v) { this._x = v ?? default; }`) instead of a plain `@Input() x = default;` field — any other component that's *both* routed *and* embedded elsewhere needs the same treatment.

### Domain model notes
- `Appointment` can optionally link to a `TreatmentPlan` (multi-session treatments, with `session_number`) and to a `Consultorio` (room). `Treatment` and `Invoice` each have an optional 1:1 back-reference to the `Appointment` that produced them. `AppointmentStatus` has 6 values: `scheduled`, `confirmed`, `in_progress`, `completed`, `cancelled`, `no_show`. **`completed`, `cancelled`, and `no_show` all free up the doctor and consultorio** — all three are excluded from availability conflict checks. Backend: `check_doctor_availability()` / `check_consultorio_availability()` in `routes/appointments.py` and the `/appointments/availability` endpoint each use `.not_in([CANCELLED, NO_SHOW, COMPLETED])`. Frontend: `appointment-form.component.ts` does **not** use the `/appointments/availability` endpoint for the doctor check — both doctor and consultorio conflicts are detected via a single `getAll(all=true, per_page=200)` call filtered client-side (status exclusion list `['cancelled', 'no_show', 'completed']`). This means availability display works without a backend restart after code changes, and both checks are always consistent. `AppointmentService.checkAvailability()` remains in `api.service.ts` (the backend endpoint still exists) but is no longer called by the form.
- `Invoice.recalculate()` is the single source of truth for `subtotal`/`total`/`balance`/`status` — call it after mutating `items`, `discount`, or `amount_paid` rather than computing those fields inline. `InvoiceStatus.PARTIAL` is legacy (kept for existing rows) — partial payments are no longer modeled as a separate invoice status, see `PaymentPlan` for multi-installment billing instead. `recalculate()` leaves `CANCELLED` alone (returns early) instead of deriving status from balance — until 2026-06-22 it didn't, so setting `status=cancelled` via `PUT /invoices/<id>` got silently reverted back to `pending`/`paid` by the very next `recalculate()` call in the same request; if you touch this method again, re-verify a cancel actually sticks by `GET`-ing the invoice back afterward, not just trusting the response of the call that cancelled it.
- An invoice can only be cancelled (`PUT /invoices/<id>` with `status: cancelled`) or have its `items` edited while `status == pending` — both are blocked once `paid`/`cancelled`. Cancelled invoices are excluded from `total_invoiced`/`total_collected` in `GET /billing/summary` and from `monthly_revenue` in the main dashboard (`pending_balance`/`monthly_pending_balance` were already status-filtered and needed no change). The frontend's invoice-detail page has a "Cancelar Factura" button (top-right, pending only) and an "Editar" toggle on the items table (pending only) that switches it to the same item-row form as the create-invoice page.
- `PaymentMethod` only offers `cash`/`qr` for new payments (`ALLOWED_PAYMENT_METHODS` in `routes/billing.py`) — `card`/`transfer`/`other` are legacy values kept solely so old `Payment`/simulated-data rows still load; don't resurrect them in any new UI.
- `PaymentPlan` cuota payments are variable amounts, not fixed to `installment_amount` (the patient can pay whatever they want, up to the remaining balance) — `POST /payment-plans/<id>/installment` accepts any positive `amount` up to `balance`. Completion (`status → completed`) is driven by `balance <= 0` (i.e. `total_paid >= total_amount`), not by `paid_installments >= installments`, since a large one-off payment can finish a plan in far fewer payments than originally scheduled. Each payment (plus the initial down payment, if any) is logged as its own `PaymentPlanInstallment` row (`payment_plan_installments` table) — `GET /payment-plans/<id>/installments` returns that history, most recent first. Plans created before this table existed (2026-06-22) have no per-payment rows for their prior progress; that endpoint detects the gap (`total_paid` > sum of logged rows) and prepends a synthetic "Pagos registrados antes de este historial" entry for the difference so the list always sums to `total_paid` — don't try to backfill real rows for that gap. A plan can only be edited (`PUT /payment-plans/<id>`) while `status == active`; patient/treatment-plan are immutable after creation. The billing list page's plan cards link to `Ver`/`Editar` (`/billing/payment-plans/:id` and `.../edit`) rather than embedding any payment-registration UI inline on the card itself — registering a payment also happens on the detail page, not the card.
- `Patient.medical_history` and `Patient.odontogram` are untyped JSON columns (free-form clinical data / per-tooth state), not normalized tables.
- `TreatmentImage` (clinical photos, FCLI-10) attaches to a `Treatment` and/or a `TreatmentPlan`; only metadata lives in Postgres — bytes live in a private Supabase Storage bucket (`app/utils/storage.py`, REST API via `requests` + the service-role key) and are only ever served back through the authenticated `GET /treatments/images/<id>/file` endpoint, never a public bucket URL. `_persist_image()` in `routes/treatments.py` wraps the upload call in `try/except storage.StorageError` and logs the real exception (`current_app.logger.exception`) before returning a generic 502 — keep that logging if you touch this path, it's the only way a Supabase Storage misconfiguration (wrong bucket, bad key, even a stray whitespace character pasted into a Render env var — this has happened) is visible from outside the container. Uploading/deleting images requires `medical_staff_required` (admin/doctor/assistant); editing a `Treatment`'s own clinical fields (`PUT /treatments/<id>`) requires the stricter `doctor_or_admin_required` — an assistant can attach a photo to an atención but can't edit its clinical note.
- `Appointment.treatment_plan_id`/`Treatment.treatment_plan_id` are denormalized into `treatment_plan_name` in each model's `to_dict()` (2026-07-06) so appointment/treatment tables (list, dashboard, patient-detail tabs) can show a clickable "Plan" column without a separate lookup. List/detail/calendar endpoints in `routes/appointments.py` and the treatments list eager-load `joinedload(*.treatment_plan)` to avoid N+1 — add the same `joinedload` to any new endpoint that serializes a list of appointments/treatments. `appointment-form.component.ts` lets the user optionally pick an active `TreatmentPlan` + `session_number` for the scheduled patient; editing an appointment keeps the plan's current value selectable even if that plan is no longer `active`, so an existing selection is never silently dropped from the dropdown.
- **Recetario estructurado** (FCLI-11, 2026-07-08): `Treatment.has_prescription` (bool) + `Treatment.medications` (JSON array of `{name, concentration, form, quantity, dosage, duration, indications}`) + `Treatment.prescription_notes` replaced the old free-text `prescriptions` textarea. `prescriptions` is kept read-only as a fallback for treatments created before this feature (`treatment-detail.component` only shows it when `medications` is empty). Validation (both frontend and `POST`/`PUT /treatments`) requires `name` + `dosage` per medication; everything else is optional. The duration field is a fixed dropdown (1–7 días + "Otro" → free text), not a new catalog table — same "Otro" escape hatch pattern used elsewhere in this codebase.
- **Clinic contact/branding fields**: `Clinic.address`/`phone`/`email`/`logo_url` (added across FCLI-11 and its 2026-07-09 follow-up) are edited only from `admin-frontend` (`clinic-detail.component`) — there is no clinic self-service profile page in `frontend/` yet (deliberately deferred, not built). `GET /api/clinic/info` (`routes/clinic.py`, `clinical_access_required`, self-scoped by `g.clinic_id`) is the only way `frontend/` reads these fields, and deliberately excludes subscription/billing fields, which stay behind `/api/platform/*`.
- **Clinic logo storage** reuses the FCLI-10 private-bucket pattern (no new bucket, no new env var): one stable path per clinic, `clinic_{id}/logo.jpg`, uploaded via `POST /api/platform/clinics/<id>/logo` (`platform_admin_required`), which always overwrites the previous logo at the same path (`x-upsert` — no orphaned files, no delete endpoint needed). `Clinic.logo_url` stores that internal bucket *path*, not a public URL — it's served only through two authenticated endpoints sharing `_serve_clinic_logo()` (`routes/clinic.py`): `GET /api/clinic/logo` (self-scoped) and `GET /api/platform/clinics/<id>/logo` (id-scoped, for admin-frontend). **`Cache-Control` deliberately differs between them**: the id-scoped route caches (`max-age=3600` — safe, the clinic id is in the URL), but the self-scoped route sends `no-store` (fixed 2026-07-09) — its URL is constant regardless of which clinic the caller belongs to, so caching it could serve one clinic's logo to a different clinic's user on a shared browser profile after an account switch. Any future self-scoped route whose response varies by tenant but whose URL doesn't must default to `no-store`, not copy the id-scoped route's caching.

### Printable documents (recetario, historia médica)
Two standalone printable views share one architecture, extracted during FCLI-15 (2026-07-09) from the recetario built first in FCLI-11:
- `frontend/src/app/shared/styles/print-document.css` — shared `@page`/`@media print` rules and base typography/spacing, imported via Angular's multi-file `styleUrls` from each print component's own CSS.
- `frontend/src/app/shared/components/print-clinic-header/` — standalone `<app-print-clinic-header [clinic] [documentTitle] [issuedDate]>` rendering the clinic's logo/name/address/phone plus a document title/date. Both `treatment-receta.component` ("Imprimir receta" button in `treatment-detail.component`) and `patient-medical-history-print.component` (print button in `patient-detail.component`'s "Historia Médica" tab) use it instead of a hardcoded header.
- Both print routes live **outside the `LayoutComponent` shell** in `app.routes.ts` (`treatments/:id/receta`, `patients/:id/historia/imprimir`) so `@media print` never tries to print the sidebar/nav — same reasoning as commit `35d765d`. Both are opened with `window.open(url, '_blank')`, never `routerLink` navigation, and are deliberately **not** `embedded` (no `@Input()` from a parent) — this sidesteps the `withComponentInputBinding()` routed-component gotcha above entirely rather than adding a third `@Input()` to something already exposed to it.
- `frontend/src/app/features/patients/odontogram-data.ts` extracts the tooth-status palette/FDI quadrant arrays/tooth names out of `odontogram.component.ts` so the interactive editor and the new `odontogram-print.component` (presentational-only SVG — no click handlers, no edit panel, takes `@Input() odontogram` instead of fetching its own) can never visually diverge.
- Both documents resolve their data in parallel (`forkJoin`) from the route id — patient/`medical_history`, odontogram, treatments (ordered by `performed_at` desc client-side), and `ClinicService.getInfo()` for the header. Empty sections (no antecedentes/odontograma/atenciones, blank contact fields) render a "sin registros" fallback or hide the field entirely via `@if` — there's no shared empty-state component, each print component handles it inline.
- No PDF-generation library anywhere — both rely on the browser's native `window.print()` / "save as PDF", an explicit decision in both design specs.

### Timestamps & timezones
Fixed 2026-07-10. Datetime fields fall into four classes — **don't lump them together**, that's exactly the bug that originally shipped (instants displaying hours off, date-only fields displaying a day early, both from calling `new Date(isoString)` on backend output with no UTC marker):
1. **Instant** (`db.DateTime`, `default=utcnow` — `created_at`, `performed_at`, `payment_date`, etc.): real value is UTC. Backend serializes with `app/utils/serialization.py`'s `iso_utc()` (appends a literal `Z`) instead of bare `.isoformat()`. Frontend then just does `new Date(iso)` — correct once the `Z` is present.
2. **Calendar date** (`db.Date`, no time component — `date_of_birth`, `due_date`, `start_date`, etc.): backend serializes as bare `YYYY-MM-DD` (unchanged). Frontend must NOT do `new Date(iso)` on these (parses as UTC midnight, displays a day early in any UTC-negative timezone) — use `formatDateOnly()` from `core/util/date.util.ts` (own copy in each frontend app), which parses the `Y-M-D` parts directly instead of going through `Date`'s ISO parser.
3. **Naive-midnight date-picker field** (`Clinic.plan_started_at`/`plan_expires_at`): already displays correctly as-is (naive `new Date()` parses as local midnight = same day) — deliberately left untouched; adding `Z` here would shift it a day.
4. **Local wall-clock** (`Appointment.scheduled_at`): naive-local end to end, backed by `clinic_time.py`'s `local_now()`/`local_today()` on the backend side for comparisons — also left untouched.

When adding a new datetime column, classify it into one of these four before writing `to_dict()` — most new fields are class 1 (`iso_utc()`) or class 2 (`formatDateOnly()` on the frontend, plain `.isoformat()` on the backend). `date.util.ts`'s `formatDate`/`formatDateTime`/`formatTime` are for classes 1 and 4 only — never use them on a class-2 field.
